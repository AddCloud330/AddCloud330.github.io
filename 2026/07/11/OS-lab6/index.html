Thinking

Thinking 6.1

管道是单向通信的，每个进程只应使用一端。如果不关闭另一端，会导致：

  • 读端进程：如果它仍持有写端文件描述符，那么当写端进程关闭写端后，读端进程的写端仍然存在，管道不会产生 EOF（文件结束符），read 会一直阻塞等待数据或写端全部关闭，造成逻辑错误。
  • 写端进程：如果它仍持有读端，那么当读端关闭后，写进程可能收到 SIGPIPE 信号或 write 返回错误，但更常见的是资源浪费。
  • 资源泄漏：每个文件描述符占用内核资源，不关闭会导致进程的 fd 表逐渐耗尽。

所以，必须显式关闭不需要的一端，否则管道无法正常终止，甚至导致死锁。

Thinking 6.2

将原代码中的父子进程角色互换：

  • 父进程：关闭 fd[1]（写端），执行 read。
  • 子进程：关闭 fd[0]（读端），执行 write。

Thinking 6.3

  • 频繁切换：缓冲区小，写者很快写满，读者很快读空，导致频繁调用 syscall_yield()，进程切换开销增大。
  • 吞吐量下降：每次写入数据量受限，大块数据需要多次系统调用和调度才能完成。
  • 影响权衡：增大缓冲区可以减少切换次数，但会占用更多内存。PIPE_SIZE 较小适合教学演示，但实际系统中通常为 4KB 或更大。

Thinking 6.4

该等式在管道稳定状态下成立。但在以下时刻不成立：

  • 正在执行 pipe_close 的过程中，若 unmap 顺序是先 pipe 后 fd，那么在 unmap pipe 之后、unmap fd 之前，pageref(pipe) 已经减 1，而 pageref(fd) 未变，等式两边差 1。
  • 正在执行 dup 过程中，若先 map pipe 后 map fd，也可能出现短暂不等。

所以，等式不是原子成立的，需要结合进程切换检测（env_runs）来安全使用。

Thinking 6.5

父进程执行 dup 复制管道文件描述符，先 map 新的 pipe 页，再 map 新的 fd 页。如果在两次 map 之间发生进程切换，子进程此时调用 _pipe_is_closed 可能看到 pageref(pipe) > pageref(fd)（因为 pipe 已增加，fd 未增加），从而错误判断写端未关闭，但实际上写端已经关闭。

Thinking 6.6

在 MOS 中，单个系统调用（如 syscall_unmap、syscall_map）在内核态执行，不会被时钟中断打断，因此是原子的。

Thinking 6.7

pipe_close 原来顺序：先 unmap pipe，再 unmap fd。
改进后顺序：先 unmap fd，再 unmap pipe。

  • 写端关闭的判断条件是 pageref(rfd) == pageref(pipe)。
  • 如果先 unmap fd，则 pageref(fd) 先减少，此时 pageref(pipe) 未变，等式不成立。
  • 之后再 unmap pipe，pageref(pipe) 减少，两者仍可能相差 1，不会出现相等的情况。
  • 因此即使中间发生进程切换，读端也不会误判写端已关闭。

对于 dup 类似：先 map fd 再 map pipe，可避免 pageref(pipe) 先于 pageref(fd) 增加。

Thinking 6.8

elf_load_seg 函数调用 load_icode_mapper 加载程序段时：

  • 先根据 filesz 从 ELF 文件中读取数据到内存。
  • 如果 memsz > filesz，则剩余部分（bss 段）通过 memset 或类似方式清零。
  • 在 MOS 中，load_icode_mapper 分配页面后，会先拷贝文件内容，然后对超出文件大小的部分写 0。

这样就实现了 bss 段在磁盘不占用空间，但在内存中占据零初始化空间。

Thinking 6.9

  1. init.c 中 ENV_CREATE(user_icode) 启动 icode 进程。
  2. icode 调用 spawn 创建 init 进程。
  3. init 进程打开 /dev/console 获得一个 fd（比如 3），然后调用 dup 将 3 复制到 0 和 1，再关闭 3。
  4. init 再 spawn 出 sh 进程，由于 spawn 会映射父进程的共享页面（包括 fd 页），sh 继承了 0 和 1，它们指向 console。

因此，标准输入输出是在 init 进程中完成设置的。

Thinking 6.10

MOS 的 sh 中，所有命令（如 ls.b、cat.b）都是通过 spawn 创建新进程执行的，因此是外部命令。
Linux 中 cd 是内置命令，因为它需要改变当前 shell 的工作目录（chdir），如果 cd 作为外部命令，它只能改变子进程的目录，父进程 shell 的目录不会变，无法达到预期效果。

Thinking 6.11

  • spawn 次数为 2 次。一次 spawn 执行 ls.b，一次 spawn 执行 cat.b。
  • 进程销毁次数为 2 次。ls.b 执行完后销毁，cat.b 执行完后销毁。

二、难点分析

难点1：管道读写中的同步与调度

读者在缓冲区空时需等待写者，写者在缓冲区满时需等待读者。为了在用户态实现这种等待而不占用 CPU ，需要：

  • 使用 syscall_yield() 主动让出 CPU，切换到其他进程。
  • 在被唤醒后重新检查条件（如缓冲区是否非空）。
  • 同时使用 _pipe_is_closed 判断对方是否关闭，避免无限等待。

难点2：管道关闭判断的竞争条件与解决方法

pageref(rfd) == pageref(pipe) 表示写端已关闭。在 pipe_close 执行过程中，如果先 unmap pipe 再 unmap fd，中间状态等式可能意外成立，导致误判。需要：

  • 修改 pipe_close：先 unmap fd，再 unmap pipe。
  • 修改 dup：先 map fd，再 map pipe。
  • 在 _pipe_is_closed 中，结合 env_runs 确保两次 pageref 之间没有发生进程切换。

三、实验体会

本次实验让我对管道作为共享内存文件的本质有了直观认识，理解了环形缓冲区、读写指针协调以及用户态通过主动 yield 实现等待的机制。最关键的收获是并发场景下引用计数操作顺序的重要性：pipe_close 中先解除 fd 映射再解除 pipe 映射，dup 中顺序相反，配合 env_runs 检测进程切换，才能避免管道关闭判断的竞争条件。此外，实现 shell 的 parsecmd 过程让我理清了重定向与管道符号的解析逻辑，以及 fork 子 shell 处理管道右侧命令的模式。整体上，实验加深了我对进程间通信、文件描述符继承和用户态并发控制的理解。

四、原创说明
