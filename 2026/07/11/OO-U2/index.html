一、同步块与锁的三次迭代分析

本单元三次作业的迭代中，我经历了从使用单一锁到设计多把协同锁，再到让全局锁与局部锁精密配合的过程，对同步块和锁的运用有了更深的体会。整个设计始终遵循仅在需要守护的共享对象上设置同步块，让“房间”和“锁”精确匹配。

hw5：以共享对象为中心，采用简单锁
基本只有一种锁，即 RequestQueue 自身的 synchronized 方法或 synchronized (waitQueue) 块。
所有对请求容器的“读-改-写”操作均封装在 RequestQueue 内部，如 addRequest、getOneRequest、pickReq 等方法均为 synchronized ，保证了操作的原子性。
线程间的等待/唤醒通过 waitQueue.wait() 和 waitQueue.notifyAll() 实现，使用典型的条件等待范式 while (waitQueue.isEmpty() && !waitQueue.isEnd()) 确保线程被唤醒后重新校验条件。
这种设计能保证数据与其锁绑定，任何线程想操作数据就必须先拿到该数据对象的锁。

hw6：引入逻辑锁 SCHEDULER_LOCK
增加电梯检修请求，打破了原有的“总有6台电梯可用”的假设，当大量电梯同时检修时，调度器无可用电梯便会空转或死等。
因此，我设计了 SCHEDULER_LOCK 作为一个全局的逻辑锁。当调度器在 chooseElevator 中发现无电梯可用时，主动执行 SCHEDULER_LOCK.wait() 挂起；电梯在完成维护、结束检修或接走乘客（即调度条件发生变化）时，以 synchronized (Scheduler.SCHEDULER_LOCK) { Scheduler.SCHEDULER_LOCK.notifyAll(); } 唤醒调度器。
同步块不再仅仅保护数据本身，更要保调度决策。例如 RequestQueue.setMaintaining 方法，在维护状态从 true 变为 false 时不仅要释放自己的锁内的等待者，还要跨对象去唤醒全局调度锁上的等待线程。这要求我们对“状态变化→通知协作对象”的流程进行仔细的锁顺序设计，避免死锁。

hw7：多锁协作
双轿厢的引入要求解决同一井道内两轿厢在F2楼层可能碰撞的物理约束，这促使我设计了两类新锁：

  • F2互斥锁（F2Lock）：一把专用的逻辑锁，通过 acquire(elevatorId) 和 release(elevatorId) 管理F2楼层的独占权。 acquire 中允许 occupier != 0 && occupier != elevatorId 时等待，但若已经持有锁则直接成功，支持可重入性的安全需求。
  • Shaft状态锁（shaftStateLock）：用于处理双轿厢均无任务且共同滞留在F2时的“让路”逻辑。当空闲轿厢需要离开F2腾出空间时，整个决策+移动过程被 synchronized (shaftStateLock) 保护，保证另一个轿厢不会与此同时抢占F2。

二、调度器设计与交互分析

调度器与线程交互模式

三次作业中，Scheduler 均作为独立的线程运行，但交互方式不断丰富：

  • hw5：典型的链状生产者-消费者。InputThread 生产请求放入 mainQueue，Scheduler 线程消费并依据官方指定的电梯ID分发到各个 subQueue，Elevator 线程再从自己的子队列消费。
  • hw6：调度器开始拥有选择权（添加 chooseElevator 方法），交互变为双向阻塞。调度器在无可用电梯时通过 SCHEDULER_LOCK 挂起，而电梯线程、请求队列则会在状态变更后去唤醒它。
  • hw7：调度器进一步规划路径和均衡负载。它接收的不再只有原始 PersonRequest，还包括换乘后重新入队的 PersonWrapper。最终代码修复后，交互相较前两次更加复杂：InputThread 也开始分担部分调度职责——将检修、更新、回收等特殊请求直接投递到目标电梯的子队列，而调度器专注于人员请求的分派与换乘拆分。

调度策略适应时间、电量等性能指标

  • 时间指标（等待时间与总运行时间）：第一次作业的简单先入先出策略性能分偏低。第二次作业我实施了“最短距离+同向优先+队列负载均衡”的启发式策略。第三次进一步细化：在 findBestElevator 中按队列长度排序，并且加入了保护性阻塞，当存在繁忙电梯且最优电梯队列长度已达到4时，调度器主动拒绝分配并等待，避免单电梯过载导致系统整体响应时间激增。这有效避免了高并发下的超时风险。
  • 电量指标：双轿厢本身就意味着电量消耗的优化机会。我的策略中，对需要穿越F2的请求，若分配给了活跃的双轿厢电梯，则通过 PersonWrapper 设定换乘标识（needsTransfer = true），将其拆分成两段独立旅程，利用下行轿厢和上行轿厢分别完成。虽然单次请求拆分会增加一些调度开销，但减少了单部电梯的大范围空驶，从全局耗电量上看是更优的。
  • 指标间的权衡：追求极致时间性能可能导致某些电梯超负荷而其他电梯闲置，增加电量消耗；过度追求负载均衡可能使某些请求被频繁等待，延长单客等待时间。我的最终策略是在负载较轻时采用最短队列优先加速响应，而在负载饱和或部分电梯繁忙时主动限制入队速率，在时间与电量之间寻找一个合理的折中。

三、程序Bug分析及多线程Debug方法

Bug1：hw6互测——检修引发的单电梯过载超时
五台电梯同时维护时，剩余一台电梯的局部队列迅速膨胀，导致超时风险。
问题出在 chooseElevator 的可用电梯筛选逻辑：当大量电梯处于 isMaintainingOrUpdating 状态时，可用列表仅剩1~2台电梯，但这些电梯的队列已经被满足，调度器依然将新请求塞入，没有阻塞。
为了修复该问题，我在 findBestElevator 中增加逻辑：若存在繁忙电梯且最优电梯的队列长度 ≥4，返回 -1，强制调度器在 SCHEDULER_LOCK 上等待。这样当仅剩的可用电梯负载较高时，新请求会被暂存，待有电梯结束维修腾出容量再分配，成功避免了TLE。

Bug2：hw7强测——F2互斥锁过早释放导致碰撞
强测中评测机判定两轿厢在F2发生“薛定谔碰撞”，即物理上应互斥的区域出现了同时占用，弱测和中测均未暴露。
经过排查发现，原始代码在电梯移动前就释放了F2锁，而移动通过 Thread.sleep 模拟，在睡眠期间轿厢逻辑上还未离开F2，但锁已被释放，给另一个轿厢提供了进入F2的机会。
为了修复该问题，我将 release 调用延迟到 move() 方法末尾，确保只在 currentFloor != 2 时才执行 f2Lock.release(elevatorId)。这样一来，锁的持有区间完整覆盖了“准备进入F2 → 在F2停留 → 离开F2”的整个物理过程，同步块内的语句与物理状态的变化严格同步，彻底消除了碰撞可能。
值得一提的是，采用上述修复后，一个原本通过的测试点因回收请求结束时间过晚而失败，且本地测试该测试点时未复现此问题。经过再次排查，我推测原因是回收请求依然走全局调度器，经过漫长等待才被执行。最终通过改造 InputThread，将维修、更新、回收等特殊请求直接放入目标电梯的子队列（绕过全局调度器），避免了不必要的排队延迟，使整体时序恢复正常。

多线程Debug方法总结
我感到多线程debug非常不便，此过程中我主要采用了大模型辅助。在出现bug时我首先考虑每个功能的单元测试，若未能定位bug，则再考虑构造极端场景（如所有电梯同时检修、所有乘客请求全部换乘）以及一些高并发场景进行压力测试，大模型可以辅助检查输出的正确性，以及针对错误信息定位代码中的问题。

四、线程安全与层次化设计的理解

线程安全
本单元让我深切体会到，线程安全的类必须做到自己管好自己，不给调用者添麻烦。以 RequestQueue 为例，它对外暴露的所有方法都是 synchronized 的，这意味着：

  • 任何线程在调用前不需要额外加锁。
  • 方法内部组合了检查和操作（如 hasPickableReq → pickReq），尽管这两个方法各自是原子的，但若外部不加锁地先后调用，仍可能产生漏洞。因此我把 pickReq 设计为“检查并取走”的原子方法，让共享对象自己守住整个过程。
  • 使用 wait/notifyAll 时永远使用 while 循环而非 if，因为从 wait 返回后条件可能已被其他线程改变。

层次化设计
三次作业的架构演化过程中，始终保持整体结构，细化变化点。

  • 稳定层：MainClass 组装线程和队列的模式始终未变；RequestQueue 作为共享缓冲区的角色不变；Elevator 的顶层处理循环（策略→执行）不变。
  • 变化层：第二次作业的变化（检修、调度选择）主要通过扩展 Scheduler 和 RequestQueue 的状态字段实现；第三次作业引入双轿厢和换乘，变化被局限在 Elevator（移动边界、F2锁）、Scheduler（换乘拆分、双轿厢激活状态）和新增的 PersonWrapper 类。Strategy 虽然每次都调整了开门与移动的决策，但对外接口 getAdvice 始终保持一致。
  • 当第三次作业需要在双轿厢井道内防止碰撞时，我只需要新增 F2Lock 和 shaftStateLock ，无需重构整个调度框架。


五、大模型使用心得

使用模型：Gemini，DeepSeek-R1

分工方式及大模型的优势：

  • 在作业初期，我会要求它解读指导书要求，分析不同架构的优劣，或者根据我的已有代码给出迭代方案。
  • 在实现阶段，遇到困难无法推进时，我会向它描述我遇到的问题，它通常会给出建议添加的方法或类。
  • 测试阶段大模型可以帮我构造复杂的测试用例。遇到bug时向它提供错误信息，它也能帮我快速定位问题所在。

遇到的困难：

  • 随着指导书内容和项目代码行数增加，无法一次性将全部内容放入对话窗口，大模型容易丢失之前讨论过的约束，需要反复提醒。
  • 模型debug有时会无中生有，将原本不需要修改的地方判错，给出的修复方案没有必要，甚至会引入新的问题。在优化时，大模型给出的优化方案也有可能是负优化的，而且由于优化时复杂度提升，大模型无法驾驭，可能引入新的bug。

六、真实体验

电梯单元是整个OO课程最声名远扬、最让人印象深刻的部分。第一单元锻炼的是形式化表达和化简能力，而第二单元直接把我们扔进了一个没有确定执行顺序的世界。当电梯开始并行运转，看到自己设计的复杂的多线程的程序能够运转时，我感到非常有成就感。但同样的，我在强测中也得到了惨痛教训，一个锁的提早释放就可能导致整个程序的失败，这种微妙的因果链也让我对“软件可靠性”有了切肤之痛。经过这三个作业，我不仅学会了 synchronized、wait、notify，更深切理解了“同步是成本，但不正确的同步是灾难”。
