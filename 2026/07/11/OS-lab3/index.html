Thinking

Thinking 3.1

这行代码实现了页目录自映射。

  • e->env_pgdir 是当前进程的页目录基地址（内核虚拟地址）。
  • PDX(UVPT) 表示虚拟地址 UVPT 的页目录索引
  • PADDR(e->env_pgdir) 获取该页目录表所在的物理地址。
  • PTE_V 是“存在位”，表示该页表项有效。

整行代码表示将页目录表自身的物理地址，作为页表物理地址填入页目录的第 PDX(UVPT) 项。这样，当 CPU 翻译 UVPT 区域的虚拟地址时，页目录项指向页目录本身，用户态程序就能以只读方式直接读取 UVPT 地址来查看自己的页表内容，无需陷入内核。

Thinking 3.2

data 参数来源于 load_icode 中的 e（当前进程的 struct Env *），通过 elf_load_seg 的最后一个参数传入，最终在回调时作为 load_icode_mapper 的第一个参数。

作用：load_icode_mapper 通过 data 获取该进程的 env_pgdir 和 env_asid，从而知道把物理页插入到哪个进程的页目录以及使用哪个 ASID 进行映射（page_insert(env->env_pgdir, env->env_asid, ...)）

不可以没有这个参数。因为回调函数是通用的，而 elf_load_seg 本身不知道当前在为哪个进程加载。如果没有 data 这样的上下文参数，映射函数就无法访问进程特有的信息（页目录、ASID 等），也就无法将页面正确挂载到目标进程的地址空间。

Thinking 3.3

elf_load_seg 需要处理这些情况：

  • 段起始的虚拟地址和文件偏移通常不是页对齐的。需要首页从页内偏移处开始拷贝数据，不覆盖该页起始处可能无关的区域。

  • 内存大小大于文件大小（BSS 段），即 p_memsz > p_filesz 。超出文件数据的部分仅分配物理页并清零，不拷贝文件内容。

  • 一个段可能占用多个页面。需要逐个页面处理，循环遍历段覆盖的所有虚拟页，对每个页面分别建立映射并加载内容。

  • 段在最后一页可能只占用部分空间。计算该页实际应拷贝的字节数，只覆盖有效数据，剩余部分清零。

  • 不同段需要不同的读/写/执行权限。需要将段头中的 p_flags 转换为页表权限标志，并通过回调参数传递给映射函数。

Thinking 3.4

env_tf.cp0_epc 存储的是虚拟地址。进程运行时，CPU 通过 MMU 将虚拟地址转换为物理地址。

Thinking 3.5

0、1、2、3 号异常处理函数的汇编包装都在 kern/genex.S 中，C语言核心逻辑在 traps.c,tlb.c,syscall_all.c 中。

Thinking 3.6

时钟中断开启的时刻：

  • 异常/中断返回
  • 首次运行用户进程
  • 用户进程正常运行期间

关闭的时刻：

  • 异常/中断入口
  • 整个内核处理过程
  • 进程调度期间
  • 内核启动早期

Thinking 3.7

硬件定时器中断 → 异常入口保存现场 → 中断处理函数判断并调用调度器 → 调度器选择新进程 → 恢复新进程上下文并返回用户态。

实验体会

Lab3 的实验内容主要是进程管理，涉及进程的创建与初始化、调度以及中断异常等知识。实验过程进一步巩固了我们对内存管理的理解（尤其是区分内核态与用户态），也让我对进程管理的整体流程与实现细节有了更深入的体会。不过在中断异常相关的汇编代码部分，我仍感到有些理解困难，今后需要加强学习，查漏补缺。
