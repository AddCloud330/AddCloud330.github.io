Thinking 1.2

尝试使用我们编写的 readelf 程序，解析之前在 target 目录下生成的内核 ELF 文件。

也许你会发现我们编写的 readelf 程序是不能解析 readelf 文件本身的，而我们刚才介绍的系统工具 readelf 则可以解析，这是为什么呢？

因为我们在 Lab1 中编写的 readelf 程序通常是针对 MIPS 架构下的 ELF32 进行设计的。在代码实现中，我们使用的结构体（如 Elf32_Ehdr）和指针偏移量都是基于 32 位定义的。
而实验环境（x86_64 Linux）默认编译生成的 readelf 可执行文件是 64 位的。64 位 ELF 文件的头部结构大小、字段位置与 32 位完全不同，因此我们的程序在解析 64 位文件时会因为格式不匹配而无法读取，甚至直接报错退出。

尝试使用 readelf -h，并阅读 tools/readelf 目录下的 Makefile，观察 readelf 与 hello 的不同。

hello 编译时指定了 -m32 参数，使其为 32 位格式，从而能被我们的程序识别；而 readelf 编译时按系统默认的 64 位生成。

Thinking 1.3

在理论课上我们了解到，MIPS 体系结构上电时，启动入口地址为 0xBFC00000（其实启动入口地址是根据具体型号而定的，由硬件逻辑确定，也有可能不是这个地址，但一定是一个确定的地址），但实验操作系统的内核入口并没有放在上电启动地址，而是按照内存布局图放置。思考为什么这样放置内核还能保证内核入口被正确跳转到？（提示：思考实验中启动过程的两阶段分别由谁执行。）

在实验中，QEMU 模拟器承担了 bootloader 的角色。当使用 make run 启动 QEMU 时，QEMU 会直接加载 ELF 格式的内核文件，根据 ELF 文件中的程序头信息将内核的各个段（如 .text、.data、.bss）加载到内存布局图中指定的虚拟地址（例如 kseg0 区域），然后跳转到 ELF 头中记录的入口地址（即 _start）。因此，内核不需要放在硬件的上电启动地址 0xBFC00000，因为 QEMU 已经完成了传统 bootloader 的两阶段工作（硬件初始化、加载内核），使得内核能够被正确加载并执行。

Exercise 1.2

难点在于理解Linker Script，并使用它控制内核加载地址。

首先我们需要确认加载地址，根据指导书，内核的.text、.data、.bss段应该加载到kseg0中。查询include/mmu.h中的内存布局图可以发现，地址0x80020000为KERNBASE，它就是.text段的起始位置，.data段和.bss段依次放在.text段后面。

实验体会

在lab1的学习中，我们离操作系统更近了一步，学习了elf文件，将内核加载到内存，启动操作系统还有printk等内容，虽然练习题工作量不大，但理解难度较lab0有了显著提高。在学习过程中，我初读指导书感到非常难懂，整体脉络也很不清晰。但是在完成练习题和思考题之后，再回过头去阅读指导书，感觉理解程度已经加深了不少。
