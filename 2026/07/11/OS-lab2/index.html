Thinking

Thinking 2.1

编写的 C 程序中指针变量中存储的地址以及 MIPS 汇编程序中 lw 和 sw 指令使用的地址都被视为虚拟地址。

Thinking 2.2

  1. 从可重用性的角度，阐述用宏来实现链表的好处：

    • 类型无关：宏在预编译期进行文本替换，不绑定具体数据类型。它像模板一样，能让同一套逻辑处理任何结构体，避免了为不同数据类型重复编写代码。
    • 侵入式设计的灵活性：通过在业务结构体中嵌入 ENTRY，使对象可以同时存在于多个数据结构中，无需修改数据结构源码。
    • 零内存管理开销：宏只操作指针，不负责内存分配。开发者可以自由决定数据存储在栈、堆还是静态区。这种解耦让链表逻辑能适配各种内存管理环境。
    • 逻辑隐藏与接口统一：将复杂的二级指针操作（如 le_prev 指向指针的指针）封装在简单的接口（如 LIST_REMOVE）下。开发者无需重复发明轮子，降低了因手写指针逻辑导致的 Bug。
    • 执行效率高：宏在调用处展开，消除了函数调用的压栈和跳转开销，且利于编译器进行指令优化。

  2. 单向链表与循环链表与双向链表，在插入与删除操作上的性能差异：

操作类型 单向链表 双向链表（本实验） 循环链表
已知节点后插入 O(1) O(1) O(1)
已知节点前插入 O(n) O(1) O(1)
删除头节点 O(1) O(1) O(1)
删除任意节点 O(n) O(1) O(1)

结论：双向链表和循环链表在任意位置插入/删除性能上优于单向链表（尤其是删除和“前插入”操作）。循环链表相比普通双向链表，额外支持反向遍历且无端部特殊判断，但实现略复杂。

Thinking 2.4

  1. ASID的必要性：
    不同的进程可能会使用相同的虚拟地址来指向不同的物理地址，引入 ASID 后，TLB 能够同时存储多个不同进程的映射项，并在查找时将虚拟地址与 ASID 结合匹配。ASID 充当了虚拟地址的扩展，使得 <虚拟地址, ASID> 这一组合在整个系统中具有唯一性，从而让硬件能够区分不同地址空间中相同的虚拟地址。

  2. 4Kc 处理器中的 ASID 字段宽度为8位，所以 4Kc 中可容纳不同的地址空间的最大数量为256。

Thinking 2.5

  1. tlb_invalidate和tlb_out的关系：
    它们之间是调用者与被调用者的关系。

    • tlb_invalidate 是 C 语言层面的封装函数：它负责逻辑上的预处理。它接收 asid（进程标识）和 va（虚拟地址），通过位运算将它们拼接成符合 MIPS 寄存器结构的 EntryHi 格式，然后将其作为第一个参数（放入 a0 寄存器）传递给 tlb_out。
    • tlb_out 是汇编语言实现的底层函数：它被 tlb_invalidate 调用，负责执行具体的硬件指令（如 tlbp, tlbwi）来操作 CP0 寄存器和 TLB 硬件表项，完成实际的擦除动作。

  2. 概括 tlb_invalidate 的作用：
    根据指定的虚拟地址和地址空间 ID，从硬件 TLB 表中查找并清除对应的映射项，以确保缓存与内存页表的一致性。

Thinking 2.6

在39位三级页表系统中，页面大小4KB，每级页表有512个条目（9位索引），虚拟地址空间为512GB（2^39字节）。采用自映射技术，通常将顶级页目录的最后一项（索引511）指向自身，以实现通过虚拟地址访问页表结构。

  1. 三级页表页目录的基地址（虚拟地址）：
    自映射下，页目录自身的虚拟地址由顶级索引、次级索引和第三级索引均为511，偏移为0得到，即
    [
    \text{虚拟基地址} = 511 \times 2^{30} + 511 \times 2^{21} + 511 \times 2^{12} = 0x7FFFFFF000
    ]
    该地址位于虚拟地址空间的最高4KB页。

  2. 映射到页目录自身的页目录项（自映射）：
    页目录中第511项（索引511，最后一个条目）指向自身，其物理地址为 (\text{PTbase} + 511 \times 8)，内容为 (\text{PTbase})（页目录的物理基地址）。

Thinking 2.7

在 MOS 操作系统中，函数调用与 CPU 访存流程的关系为：
当 CPU 访存遭遇 TLB 缺失或页面不存在等异常时，硬件会通过异常机制跳转到内核函数（如 do_tlb_refill 或 passive_alloc），由这些软件函数完成物理页分配（page_alloc）、页表建立（page_insert）以及硬件寄存器更新（tlb_out），从而可以返回重新访存，这个过程中通过 tlb_invalidate 等函数确保了软件映射与硬件缓存之间的一致性。

Thinking 2.8

  1. X86 内存管理机制简述
    X86 采用的是分级管理机制，包含“分段”和“分页”两个过程：

    • 分段：将逻辑地址转换为线性地址。通过段寄存器索引段描述符表，提供权限控制和隔离。
    • 分页：将线性地址转换为物理地址。由硬件内存管理单元（MMU）自动遍历多级页表（如 CR3 寄存器指向的页目录），并利用硬件自动维护 TLB 的重填。

  2. X86 与 MIPS 的内存管理区别

特性 X86 (CISC) MIPS (RISC)
地址转换 逻辑地址 $\to$ 线性地址 $\to$ 物理地址 虚拟地址 $\to$ 物理地址（不强制分段）
TLB 重填 硬件自动完成。MMU 自动遍历页表，无需软件干预。 软件手动完成。TLB 缺失触发异常，由 OS 内核代码填充。
页表结构 页表格式由硬件规定。 页表格式由软件（OS）定义，更灵活。
管理重心 侧重于硬件实现的复杂度和自动化。 侧重于简化硬件设计，将复杂度移交给软件。

难点分析

Exercise 2.1-2.5

这几个习题涉及物理内存管理，分别是补全include/queue.h 中的 LIST_INSERT_AFTER 宏和 kern/pmap.c 中的 mips_detect_memory,page_init,page_alloc,page_free 函数。
难点主要在于

  • 理解Page结构体、Page_list结构体的结构以及各变量的含义
  • 理解物理内存管理的原理，区分Page结构体指针和对应物理页地址的用法，掌握两者间的映射关系
  • 需要掌握C语言指针和数据结构的知识，理解并熟练运用链表宏

Exercise 2.6-2.10

这几个练习涉及虚拟内存管理和TLB。
首先需要我们理解两级页表结构的地址变换机制。这个过程中需要注意区分物理地址和虚拟地址：访问页目录和页表需要使用虚拟地址，页目录项和页表项中取出的是物理页号。此外，还需要掌握通过位运算方法来提取或设置权限位。

而且，TLB 重填的实现过程理解难度较高。不仅要通过 CP0 寄存器手动维护 TLB 映射，还需在 _do_tlb_refill 中利用位运算处理奇偶页对齐与 PFN 格式转换。

实验体会

通过学习Lab2的内容，我对内存管理的过程有了更加细节和深刻的体会。一开始，我使用链表宏时很不熟练，而且在访问和操作页表时，物理地址、虚拟地址和物理页对应的page结构体，三者的转换让人晕头转向，此外TLB重填的过程涉及到异常，更加复杂。而且，lab2的整体内容相比lab0和lab1也有所增长。但是通过理解和练习，从一头雾水到基本理解，我对内存管理这一重要部分有了更多认识，也认识到了自己理论知识的不足，还需要更多的学习。
