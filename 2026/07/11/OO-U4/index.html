一、正向建模与两阶类图

本单元最大的改变不是新增了多少需求，而是开发流程本身被重新定义了——先画图，再写代码；一阶类图是设计承诺，是最初的构想，二阶类图则需要根据实现情况进行设计修正，两者之间必须有可追踪的演化路径。

1. 一阶类图

一阶类图在代码之前提交，那时对需求的理解往往还不完整，画出的一阶类图也不够具体、不够完整。但是它帮助我们对整体有一个把握，迫使我们在信息不充分时做出设计决策。

对我来说，一阶类图的核心作用是建立骨架。在 U4 中，我首先识别出六个核心地点类（Bookshelf、TreasuredBookshelf、BorrowReturnOffice、AppointmentOffice、ReadingRoom、GradeManager）以及它们与 LibraryManager 的聚合关系。这些类的类名直接来自需求中的名词，几乎不需要创造性思考，但把它们画出来之后，系统的边界就清晰了。

而且，一阶类图可以提醒我现在还有哪些细节是不清楚的、需要进一步注意和确认的。比如在第一次画图时，我对 AppointmentOffice 中预约图书的过期逻辑只有模糊概念，类图中的 removeExpired 方法也没有填上参数。此时我可以仔细重读需求中关于“开馆前整理送达保留5天，闭馆后整理送达从次日起算”的细则，在编码之前就处理这些细节；也可以暂时不处理这一细节，在二阶类图中再做细节补充。

2. 二阶类图

二阶类图的提交在编码完成后，评测会检查它与代码的一致性（R1-R5），并与一阶类图做相似度比较。这意味着我们不能把一阶类图当废稿扔掉，二阶类图也不是从头重画一遍，而是展示设计如何随着对问题的深入理解而演进。

这些变化如果只存在于代码中，一阶与二阶的差异就无法被追踪。而有了二阶类图和重构说明，整个设计演进过程是透明的，这也是两阶类图机制最核心的设计意图。

3. 类图与代码的追踪关系

U4 代码中，类图与程序的追踪关系可以清晰地梳理出来：

类图中的设计单元 代码中的对应实现 追踪方式
Shelf 接口 明确的 interface Shelf 接口定义
Bookshelf implements Shelf class Bookshelf implements Shelf 实现关系
LibraryManager 聚合六个地点类 六个 private final 成员变量 关联关系

R5 评测要求类图中的继承/实现/关联关系与代码保持一致，因此我在实现过程中，每当新增一个类或修改一个关系，都会同步更新类图，保持了设计与实现的一致性。

4. 两阶类图的价值

经过这个单元的三次作业，我体会到，两阶类图不是为了给评测增加难度，而是为了引导我们通过思考设计架构，一阶类图提交时，我们可能对需求的理解还不完整，但是有一个起点。二阶类图记录了我们从起点到终点的路径，重构说明解释了为什么偏航。整个过程中，类图使得我们的开发流程更加明晰。

1. 整体架构

2. 核心设计决策

地点即状态容器。每个地点类本质上是 Map<Book, ...> 或 List<Book> 的封装，职责单一：Bookshelf 只管存书取书，AppointmentOffice 只管预约书的到达、领取和过期清理。这种设计让每个类的代码量控制在合理范围内，也方便了整理流程中的批量操作。

Book 持有自己的轨迹。Book 类内部维护了 List<LibraryTrace>，每次状态转移时追加记录，让数据自己管理自己。查询历史轨迹时直接从 Book 读取，不需要从各个地点类反向收集。

LibraryManager 只做编排，不做具体操作。借书流程中，LibraryManager 负责检查用户权限、从书架取书、更新用户持有列表、记录轨迹，但具体的“从书架移除”操作由 Bookshelf.removeBook 完成，“添加轨迹”由 Book.moveToUser 完成。控制器只负责流程控制，不越界操作。

3. 状态图与代码的对应

U4 中状态图的评测引入了 @Trigger 注解机制，要求程序中的状态转移方法与状态图的迁移一一对应。我的 Book 类中，每个 moveToXxx 方法都标注了 @Trigger(from = "...", to = "...")，状态图也从 InitState 出发覆盖了所有可达状态。这种设计使得图书的生命周期（书架 → 用户/阅览室/预约处 → 借还处 → 书架）在设计和实现层面都清晰可溯。

三、引导大模型辅助正向建模的策略

四个单元中我都使用了大模型辅助开发，但不同单元的用法有显著差异。在 U4 中，大模型在正向建模方面的作用非常明显，但也有一些局限。

1. 有效的引导策略

先画框架，再填细节。我不会一次性让大模型生成完整类图，而是首先描述核心实体（用户、图书、各个地点）及其关系，让大模型给出类结构和关联，然后在确认框架合理后，再逐个细化每个类的方法签名。

用需求文档中的动词命名方法。我将指导书中的“借书流程”“还书流程”等描述直接输入，要求大模型提取关键操作并映射到类的方法上。borrow() 在 LibraryManager，returnBook() 在 BorrowReturnOffice，pickBook() 在 AppointmentOffice——这些命名直接来自需求，大模型很容易给出合理的分配。

让大模型生成代码骨架，再手工调整。在确认类图后，我会让大模型生成对应的 Java 类框架（成员变量、构造器、主要方法签名），然后在此基础上手工实现业务逻辑。这大幅减少了重复性编码工作，同时保证了类图与代码的初始一致性。

2. 局限与应对

我觉得大模型在正向建模中最明显的问题是不理解业务语义的微妙之处。比如需求中“开馆前整理送达保留5天”和“闭馆后整理送达从次日起算保留5天”这两个表述，有时大模型只会把两者都翻译成“加5天”，意识不到它们在过期判断上需要用不同的逻辑（isAfter vs !isBefore）。同样，在涉及多类协作的流程中——比如预约取书要同时更新 AppointmentOffice、User、Book 三个类的状态——模型生成的代码也很难完全正确。

因此，我觉得大模型只适合做“帮助写快一点”的事，不适合做“代替想清楚”的事。凡涉及业务规则判断和跨类流程控制的部分，我们最好自己写；模型只负责生成那些标准化、无争议的辅助代码，比如 addBook、getCredit 这种增删查改方法。在类图阶段，我也不让它直接画完整的图，而是自己先画草稿，再让它根据需求文档帮我检查是否有遗漏的类或方法，这种“查漏补缺”的工作非常适合它。总之，把它当作一个快速应答的助手，而不是能够独立完成设计的架构师。

四、架构设计思维的演进

四个单元的作业，我的架构设计思维不断发展。

第一单元（基于需求扩展设计）

U1 让我学会了有意识地提高代码的扩展性。
以前，我习惯性地把所有逻辑塞进一个类，直到第二次作业新增因子种类时才意识到这种做法的代价。引入访问者模式让我第一次体会到“好的架构可以降低迭代成本”——新增 DeriveFactor 只需要新增一个类和对应的 visit 方法，原有的 ExpPoly 运算逻辑完全不需要修改。这一单元让我建立了扩展点的概念，在设计时识别哪些部分是稳定的，哪些部分会变化，然后把变化的部分抽象出来。

第二单元（从顺序转向并发）

U2 的挑战不仅是多线程语法，更是思维方式的转换。
在单线程程序中，你控制执行顺序；在多线程程序中，你只能控制“规则”。设计 SCHEDULER_LOCK 和 F2Lock 的过程中，我学会了用“状态→事件→响应”的视角来思考：当某个条件不满足时，线程应该等待什么事件；当某个事件发生时，哪些线程需要被唤醒。锁的粒度设计也变得更加精准——不是锁越少越好，而是锁的范围越精确越好。

第三单元（规格化开发）

U3 的 JML 规格驱动开发让我摆脱了“凭感觉写代码”的习惯。
规格不是约束，而是唯一可信的参考。当我对某个方法的行为不确定时，不再靠猜测或试错，而是直接读 JML 表达式。这种思维延伸到单元测试中，我学会了根据规格设计测试用例，而不是根据实现设计测试用例——前者测试“应该做什么”，后者测试“实际做了什么”，两者的边界是清晰可辨的。

第四单元（模型驱动）

U4 的正向建模把“设计”从“实现”中独立出来。
之前三个单元，我都是只写代码，最多在单元总结的时候通过类图来分析一下自己的程序。U4 要求先画图再写代码，我刚开始很不适应，因为觉得图又不能编译，也难以评判，到底有什么意义？但随着迭代推进，我意识到好的设计需要可视化的抽象。类图让我在写第一行代码之前就能评估架构的合理性——LibraryManager 是否承担了太多职责？Shelf 接口能否统一 Bookshelf 和 TreasuredBookshelf 的行为？这些问题在类图上一目了然，而如果在代码层级才发现，修改成本会高得多。

五、测试思维的演进

纵观这四个单元的作业，我的测试环节有很大的不足，也因此得到了惨痛的教训。不过，这个过程中我的测试能力也逐渐建立。

第一单元（边界构造）

U1 初期我的测试方式很简单：随机生成一些表达式，看输出是否合理。这种方式只能发现最明显的 bug，对于边界情况（如多层嵌套、指数为0）覆盖不足。第二次作业中，因为对选择表达式两个分支都进行了求值导致超时，暴露了随机测试覆盖率不足的问题。之后我开始有针对性地构造边界样例，尤其是那些会导致大量递归或复杂运算的输入。

第二单元（压力测试）

多线程程序的 bug 往往在特定时序下才出现，随机输入很难触发。我学习了用大模型生成极端并发场景——所有电梯同时检修、大量乘客同时到达、双轿厢在 F2 的抢占——这些压力测试帮我发现了 SCHEDULER_LOCK 的唤醒时机问题和 F2 锁的释放时机问题。这一阶段我认识到：多线程测试不仅要测功能正确性，还要测性能退化边界。

第三单元（单元测试）

JML 规格让测试用例的设计有了明确的依据。我为方法编写基于规格的 JUnit 测试：

  • 副作用验证：pure 方法调用前后做状态快照对比
  • 异常优先级验证：按规格中 signals 的顺序，测试不同异常场景
  • 幂等性检查：连续调用两次 pure 方法，断言结果相同

这种方法让 bug 的定位从“强测崩溃”提前到了“单元测试断言失败”，大幅降低了调试成本。

第四单元（场景交互测试）

U4 的交互式评测要求程序在整理流程中的输出会影响后续的取书请求，测试不再是单点验证，而是全流程场景测试。我的测试策略调整为：

  • 构造完整的日期序列（包括开闭馆），验证整理输出与后续请求的联动
  • 测试预约过期逻辑：精确控制开馆前/闭馆后整理的时间点，验证保留期计算是否正确
  • 测试信用分扣减：阅读不归还、预约不取、逾期还书等多个扣分场景的组合

六、课程收获

不知不觉间，OO 课程已经走到尾声，让我真是百感交集。
首先讲讲课程内容带给我的收获吧，这应该是我第一次开发规模这么大的代码，四个单元聚焦不同角度：

  • U1（表达式）：数据表示与运算的抽象
  • U2（电梯）：并发控制与资源调度
  • U3（社交网络）：契约式设计与规格驱动
  • U4（图书馆）：正向建模与模型驱动

我最深刻的感受就是：代码是实现，设计才是核心。好的设计不仅在正确性和性能上表现优越，还通过分层解耦实现了扩展性、降低了调试难度。相反的，不好的设计会让我们付出更大的工作量完成一份没那么好的程序，甚至会需要我们在后续的迭代中重构。所以，设计架构的能力实际上是非常重要的。面对新场景新需求，我们应该首先思考核心实体是什么、它们之间是什么关系、哪些部分会变化，再根据这些问题的答案决定架构的方向。

此外，我还认识到测试是一种设计态度，它不仅仅是“写完代码后的验证工作”，还是设计的一部分。规格驱动开发中，测试用例和代码同样重要；多线程开发中，压力测试是检验设计是否正确的手段。把测试当做必要的一个环节、并为此投入时间和心血的同学，必然取得更好的成绩。OO 的作业没有标准答案，只有持续改进。这种耐心地面对复杂系统的能力，或许是我从这门课中收获的、超越技术本身的东西。

在这里还想提一下大模型，它深度参与了我的 OO 征程，包括不限于帮我分析指导书、完成部分代码、检查代码、生成测试数据等工作。现在大模型的能力已经强大到令我们迷茫的地步，特别是当我发现自己在开发过程中无法控制地依赖它，我开始思考，我在 OO 课程中取得的成果真的是我自己的成果吗？所以，也许更加重要的是具备鉴别和判断的能力、掌握全局的能力。现阶段来说，AI 辅助编程已经是不可逆的趋势，也许，别的不说，我们在 OO 过程中至少学会了如何更好地使用大模型。

最后有些感慨，这学期各方面的事情都很多，OO 每周繁重的任务给我带来过很多压力和烦恼，然而，回顾时那些痛苦都仿佛已经远去，我只感到遗憾，为什么这个过程中自己没有再做得好一点。

是时候说再见了，也许 OO 课程虽然结束，面向对象设计与构造的旅程才刚刚开始，希望自己可以再度成长。
